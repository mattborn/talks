A brief history

  • JSS proposed by Netscape in 1996
  • CSS was a rival standard
  • Who ultimately decided the fate of JSS by not implementing it? Internet Explorer.
  • Why did CSS win? Because the internet was and still is mostly documents.
  • In May 2003, [CSS Zen Garden](http://csszengarden.com) convinced us this was the correct path. (but these were very basic formatted documents)
## Why Now? - [Web Components](http://webcomponents.org) - Facebook Evangelism - In Javascript We Trust
## Web Components - Good encapsulation = components only know about themselves - Community consensus = ES6 modules (next version of JS), [Polymer](https://www.polymer-project.org), AMD, CommonJS, Require, Webpack, Browserify, React … - Not a library, but an architectural design pattern
## Facebook Evangelism - We haven’t actually separated concerns, we separated technologies. - If the markup changes, we still have to change the styles. - “We believe colocating style with your JS is a good idea.” - “It turns out that if you write your styles in JS, a large class of really hard problems with CSS just disappear instantly.” - Instead of focusing on what has changed, let’s focus on what is different.
## In Javascript We Trust - We already use Javascript for events + DOM manipulation. - Javascript can create elements. - Javascript can style elements. - Javascript is an expressive language that already has everything LESS + SASS are trying to account for.

## The React Way - Why not do everything like we did in the 90s? - React says just re-render everything every time something changes, but only update what’s different. - Let’s make expressing markup native without templates + bind our events directly as well. - Hell, why not bring inline styles back like it’s 1999?

Inline styles do not represent an alternative for the systematic design, implementation and governance of consistent coding standards, tooling, or processes.
